# T-Coding

用于 T 仓库的学习型开发协作。

这个 skill 不是"写代码"，而是按照 T 仓库的组织方式，帮用户把一次学习或实验沉淀为可回溯的成果：

- 学习目标 → `todo.md`
- 一个概念 → 一个 Demo → 一个 Commit
- 当前工作区 → 最新实验
- 历史知识点 → 通过 Git 与 docs 分支回溯

## 仓库定位

T = 深度 + 广度。

它是一个 commit-driven learning repository，也是一个 coding playground，用于系统化学习与实践编程概念。

| 角色 | 说明 |
|------|------|
| Demo / MVP | 最小可运行实践 |
| Git | 时间轴与版本状态 |
| Markdown | 索引与知识结构 |
| docs 分支 | 完整索引表，通过 commit hash 回溯任意历史知识点 |

当前工作区只保留最新实验，历史通过 Git 回溯。

> 这个仓库不是代码集合，而是一条可回溯的学习轨迹。

## 何时触发

当且仅当用户明确处于 T 仓库语境，且需求属于以下场景之一时触发。

若当前工作目录就是 T 仓库，且用户请求与学习、demo、MVP、README、commit message、docs 索引或 skill 维护有关，默认视为 T 仓库语境。

1. 规划学习路径，生成或调整 `todo.md`
2. 编写某个概念对应的 demo 或某个主题对应的 MVP
3. 为当前改动生成符合仓库约定的 commit message
4. 更新 docs 分支索引或整理知识索引
5. 明确要求"以后按这个流程做"或"更新 skill / 更新规范"
6. 知识讲解、概念梳理、自主编码（用户自己写代码，AI 负责讲解和引导）
7. 将对话中的知识笔记整理写入 `README.md`

## 不要触发

以下情况不要默认触发：

**明确不问：**
1. 普通项目开发、业务需求开发、生产级工程改造
2. 通用编程问答，与 T 仓库结构无关
3. 泛泛写代码，无 Demo / MVP / 学习沉淀 / commit-driven 语境

**需要先确认：**
4. 用户在 T 仓库内做临时验证性实验，需判断按 demo 沉淀还是临时验证
5. 用户提到"学习"但完全没有可判断的主题或产物，需先确认是否进入 T 仓库流程
6. 用户在 T 仓库内做知识问答，但看不出是否要沉淀为自主编码练习时，先确认

## 工作原则

1. 优先服务"学习沉淀"，不一味追求功能堆叠
2. 默认产出最小可运行版本，避免一次塞入过多概念
3. 每次只围绕一个清晰主题推进，必要时提醒用户拆分为多个 demo
4. 学习笔记优先沉淀在源文件注释中；README 偏重概述、入口导航与运行说明
5. 如用户描述模糊，优先根据仓库上下文和用户语义合理判断；只有会影响产物形态、编号、分支或是否写文件时，才先确认
6. 若当前约定与 skill 冲突，以用户本次明确确认的最新约定为准，并在需要时回写 skill

## 开始前检查

执行前先确认以下信息是否足够明确：

1. 当前任务属于 `todo.md`、demo、MVP、commit message、docs 索引、skill 更新，还是自主编码与知识沉淀
2. 主题是否足够具体
3. 目标语言或技术栈是否明确
4. 本次产物是什么
5. 是否需要用户先确认某个关键选择

若缺失信息不会影响产物形态、编号、分支或是否写文件，可以先按最合理路径推进；若会影响这些关键选择，先问最少但关键的问题。

## 最小执行模板

执行任务时默认按以下顺序推进：

1. 识别当前属于哪项能力
2. 加载对应 reference
3. 检查关键上下文（主题、产物、技术栈、分支、编号）
4. 执行任务
5. 按对应交付检查清单自查
6. 给出必要的下一步提醒

## 核心原则

以下原则贯穿全部能力，详细规则见对应参考文档。

### 命名映射原则

`领域`、docs 主题文件名、Markdown 标题可以不完全同名，但必须表达同一个主题，并保持映射稳定。一旦某个主题已有固定 docs 文件名，后续沿用。

> 完整规则见 `references/docs-format.md`，该文件为映射规则的唯一定义源。

### 语言分工原则

- commit message 必须使用英文，优先英文官方术语
- `todo.md` 和 docs 表格中的 `知识点`、`说明` 默认使用中文
- 从 commit 映射到 docs / todo.md 时，不要机械照抄英文，应转换为自然、准确的中文
- 如果中文难以表达或容易歧义（如 "trait"、"crate"、"serde"），允许保留英文术语混合使用（如 "trait 与泛型"）

> 完整规则见 `references/commit-convention.md` 和 `references/docs-format.md`。

### 交付形态原则

默认两类形态：单文件脚本（适合语法、基础概念验证）和项目结构（适合框架、构建工具本身是学习目标的场景）。

> 完整规则见 `references/code-format.md` 和 `references/directory-structure.md`。

### 知识沉淀原则

知识笔记的载体取决于谁写代码：

- **AI 写代码（能力二）** → 知识笔记写在源码注释中，README 偏重概述与运行说明
- **用户写代码（能力六）** → 知识笔记在对话中讲解，用户验证后整理写入 README

README 在两种模式下承担不同角色，不要混用。

> 自主编码的 README 格式见 `references/self-coding-readme.md`。

## 默认产物

| 任务 | 默认产物 |
|------|----------|
| 学习路径规划 | `todo.md` |
| Demo / MVP | 可运行代码 + 必要的 `README.md` |
| Commit 消息 | 符合规范的 commit message |
| docs 索引更新 | 按用户要求更新索引内容 |
| workflow / skill 更新 | 默认更新 AI 已安装的 skill 副本；用户明确要求同步源文件时，再更新 docs 分支中的 skill 源文件 |
| 自主编码与知识沉淀 | 知识讲解（对话） + `README.md`（用户触发时从对话整理写入） |

执行具体任务前，按以下映射加载对应参考文档：

## 六大能力

### 能力一：学习路径规划 → 生成 `todo.md`

触发场景：用户有明确实战目标、想知道"为了做成 X 需要学什么"、或明确要求生成 `todo.md`。

核心步骤：
1. 提炼最终实战目标，不要只抄原话
2. 拆出达成目标所需的关键知识点
3. 按仓库节奏组织为 demo 路线，不是课程大纲
4. 跨领域/多阶段时，优先按"学习目标 → 分阶段 → 每阶段一张表"组织
5. 生成或更新 `todo.md`

> 格式与字段规范：加载 `references/todo-format.md`。

### 能力二：生成可运行的 Demo / MVP

触发场景：用户要求"帮我写 demo"、"做一个最小可运行示例"、"围绕某个知识点做实验"、或"做一个 MVP"。

与能力六的分流：
- 用户表达为"帮我写"、"做一个 demo / MVP"、"生成可运行代码"时，默认走能力二
- 用户表达为"讲讲"、"解释一下"、"我自己写"、"引导我写"时，默认走能力六
- 如果用户只说"我想学 X"，且无法判断是否需要 AI 代写代码，先确认是要 AI 写 demo，还是用户自主编码

核心步骤：
1. 明确本次只聚焦一个主题或一个最小目标
2. 判断单文件脚本还是标准项目结构
3. 如果是单文件脚本：
   - **一个知识点 = 一个 app 文件**，按序号命名：`app1.rs`、`app2.rs`、`app3.rs` ...
   - 每个文件只覆盖一个独立知识点，保持轻量可读
   - 知识笔记写在源文件注释中（AI 写代码时）
   - 所有 app 文件属于同一个 demo，用同一组 commit 编号
4. 如果是标准项目结构：按框架目录组织
5. README 只做知识罗列，用表格展示知识点分布，不与代码文件绑定
6. 提醒用户自行运行验证

> 代码风格与目录规范：加载 `references/code-format.md` + `references/directory-structure.md`。

### 能力三：生成 Commit 消息

触发场景：用户说"帮我生成提交消息"、"这次提交该怎么写"、或想校验当前 message 是否符合规范。

核心步骤：
1. 判断是新的 demo、补充提交（`dN+`/`mN+`），还是新的 MVP
2. 依据约定生成 commit message（必须英文术语）
3. 编号不确定时，协助从 `git log` 确认，不凭空编造

> 完整约定：加载 `references/commit-convention.md`。

### 能力四：更新 docs 分支索引

触发场景：用户明确要求"更新索引"、或要求将当前阶段成果整理进 docs 分支。

核心步骤：
1. 确认要更新的主题、文件和提交范围
2. 安全检查：main 工作区必须已提交干净；当前分支必须是 docs 分支
3. 若仍在 main 分支，不自动切换分支；提示用户手动切到 docs 分支，切好后继续
4. 按规范更新对应主题文件的索引表格

> 格式、映射规则与分支操作细节：加载 `references/docs-format.md`。

### 能力五：维护 skill 配置

触发场景：用户说"以后这么做"、"帮我更新 skill"、或明确提出新的约定/流程。

运行时生效的 skill 是 AI 已安装的本地副本。skill 源文件维护在 docs 分支的 `/skills/t-coding`，安装时使用文件拷贝。

核心步骤：
1. 提炼用户刚确认的最新规则、偏好和流程
2. 判断影响范围：只改主文件 / 只改 reference / 两者都改
3. 默认更新 AI 当前已安装的 skill 副本
4. 只有用户明确要求"同步源文件"、"更新 docs 分支里的 skill"或"更新仓库中的 skill"时，才同步修改 docs 分支 `/skills/t-coding` 中的 skill 源文件
5. 自查：新规则是否与其他 reference 矛盾；若有矛盾，同步修正
6. 主文件只描述总规则，细节沉淀到 `references/`
7. 若某段规则已稳定，写成"触发条件 + 步骤 + 注意事项"格式

注意：若只是某次任务的临时要求，不要误写成长期规范。

### 能力六：自主编码与知识沉淀

用户自己写代码，AI 负责知识讲解、引导验证，并将学习笔记整理写入 README。

触发场景：
- 用户说"讲讲 X"、"解释一下 X"、"我想学 X 但自己写"
- 用户明确表示不要求 AI 生成完整代码，但需要知识讲解和编码引导
- 用户说"帮我整理 README"或"把这些知识点写进 README"

核心流程（一个知识点的完整闭环）：

1. **确认范围** — AI 与用户确认主题、边界和预期验证方向
2. **知识讲解**（对话中，不写文件）— AI 在回答中讲解概念，给出说明性代码片段、知识拆解和验证建议
3. **用户编码** — 用户自行写代码验证，AI 不代笔
4. **README 沉淀**（用户说"更新 README"时触发）— AI 从对话上下文提炼知识点内容，增量追加到 README；README 只记录知识点笔记，不搬运 AI 回答模板
5. **提交** — 用户验证代码 + README 无误后，AI 协助生成 commit message

关键规则：
- 每个知识点完成**完整的"讲解→编码→README→commit"闭环**再进入下一个，不堆积
- AI 讲完一个知识点后主动提醒："写好验证代码后告诉我，我帮你把学到的内容整理进 README"
- README 是**增量追加**，不覆盖之前的知识点
- commit 时 README 和用户代码一起提交，不拆分

> README 格式与整理规则：加载 `references/self-coding-readme.md`。

## 能力流转

完成某项能力后，不自动衔接下一步，但应适时提醒用户：

| 完成后 | AI 应提醒 | 衔接到 |
|--------|----------|--------|
| 能力一生成 todo.md | "选一个知识点开始吧" | 能力二或能力六 |
| 能力二写完 demo | "运行验证后可以提交" | 能力三（用户要求时） |
| 能力六讲解完毕 | "写好验证代码后告诉我，我帮你整理 README" | 能力三（用户要求时） |
| 能力三提交完成 | 无需主动提醒 | 能力四（用户要求时） |
| 能力四更新 docs | 无 | — |

## 冷启动

若仓库刚初始化（无 commit 历史）：

1. 第一个 demo 直接用 `d1`
2. 不强制要求先创建 `todo.md`
3. docs 分支由用户手动创建，skill 不自动创建分支

## 交付检查清单

完成任务前，按当前执行的能力自查：

**能力一**：□ todo.md 围绕实战目标而非泛泛课程目录 □ 每行可映射到未来一个 demo commit

**能力二**：□ 每个 app 文件只覆盖一个独立知识点 □ 代码可运行 □ 注释包含知识笔记 □ README 为知识点表格（不绑定代码文件）

**能力三**：□ 编号从 git log 确认，不凭空编造 □ 英文术语 □ dN+/mN+ 仅用于连续补丁

**能力四**：□ main 已提交干净 □ 当前在 docs 分支 □ 映射到正确主题文件

**能力五**：□ 新规则不与其他 reference 矛盾 □ 主文件未塞入本属 reference 的细节

**能力六**：□ 知识讲解结构化 □ README 增量追加而非覆盖 □ 代码与 README 同一 commit
