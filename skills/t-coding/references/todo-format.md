# todo.md 规范

`todo.md` 用于承接当前阶段尚未完成的学习计划。

它不是课程目录，也不是任务管理软件导出的长列表，而是一份面向 T 仓库工作流的“待完成知识点索引”。

目标：

1. 帮用户拆出接下来要做的 demo
2. 让每条 todo 都能自然落到一次 commit
3. 让完成后的内容能方便迁移到 docs 分支，并覆盖到知识点的最新索引

## 与 docs / commit 的关系

`todo.md` 应尽量与最终 docs 索引结构保持一致。

字段映射如下：

| todo.md | commit message | docs |
|---------|----------------|------|
| `领域` | `领域` | `xx.md` |
| `知识点` | `学习主题` | `知识点` |
| `说明` | `具体描述` | `说明` |

其中：
1. `领域` 映射规则见 `references/docs-format.md` 的"领域映射规则"
2. `知识点` 对应未来某一次 demo commit 的标题主题
3. `说明` 对应未来 docs 表格中的"说明"，写提纲即可

## 推荐格式

`todo.md` 的结构可以灵活安排，不要求永远只有一张总表。

当学习目标会交叉多个领域时，优先按“总目标 + 分阶段章节 + 每个阶段一张表”的方式组织。单张表内部允许同时出现多个领域，只要每一行仍然能清晰映射到未来的 commit 和 docs 索引即可。

常见组织方式：

1. 单一主题学习：一个标题 + 一张表
2. 跨领域目标学习：一个总目标 + 多个阶段章节 + 多张表
3. 大目标分阶段推进：一个总目标 + 多个阶段标题 + 每个阶段各自表格

### 形式一：单表结构

```markdown
# 学习计划 - <目标名称>

> 目标：一句话说明这轮学习最终要做成什么

| 领域 | 知识点 | 说明 | 时间 |
|------|--------|------|------|
| regex | 普通字符和元字符 | 基础字符匹配、特殊元字符含义 | - |
| regex | 量词 | 贪婪与非贪婪匹配 | - |
| regex | 分组 | 分组、捕获组、命名捕获、反向引用、非捕获组 | - |
```

### 形式二：分阶段多表结构

```markdown
# 学习计划 - <目标名称>

## 学习目标

一句话说明这轮学习最终要做成什么

## 第一阶段

| 领域 | 知识点 | 说明 | 时间 |
|------|--------|------|------|
| rust | ownership and borrowing | ownership rules, borrowing, lifetimes | - |
| js | event loop | macrotask, microtask, execution order | - |

## 第二阶段

| 领域 | 知识点 | 说明 | 时间 |
|------|--------|------|------|
| rust | error handling | Result, Option, custom errors | - |
| regex | 分组 | 捕获组、命名捕获、反向引用、非捕获组 | - |
```

## 字段要求

| 字段 | 说明 |
|------|------|
| `领域` | 对应 docs 主题标识，可使用简称或别名，如 `regex`、`js`、`swift` |
| `知识点` | 当前计划中的一个可独立成 demo 的知识点 |
| `说明` | 对知识点的展开目录，简短即可；如果没有特别要写的，用 `-` 占位 |
| `时间` | 初始留空；该 demo 对应的 commit 完成后，将 commit 时间填入此列 |

补充说明：

1. 领域映射规则见 `references/docs-format.md` 的"领域映射规则"
2. 一旦某个主题已经在 docs 中采用固定文件名，后续 todo 应继续映射到同一个主题文件

## 编写原则

1. 一行最好对应未来一个 demo commit
2. `知识点` 要足够聚焦，避免一行塞入太多主题
3. `说明` 写目录和提纲，不写大段正文
4. 如果某知识点无法独立形成一个 demo，说明拆分粒度不合适
5. 如果多个知识点都落到同一个 docs 文件，可以重复相同的 `领域`
6. 若学习目标跨多个领域，优先按阶段拆成多张表，而不是先按领域拆表
7. 同一张表中允许出现多个领域，因为真正决定 docs 去向的是每一行里的 `领域`
8. docs 主索引本身没有 `领域` 这一列，因此迁移时应按每一行的 `领域` 分发到对应主题文件，而不是按 `todo.md` 的表格分组整体迁移
9. 阶段标题服务于阅读和规划；真正用于映射 commit / docs 的仍然是表格中的 `领域`、`知识点`、`说明`

## 生成步骤

当用户要求生成 `todo.md` 时：

1. 先提炼最终目标
2. 再拆成若干可独立提交的知识点
3. 判断这些知识点是否需要分阶段推进；如果是，优先按阶段组织成多张表
4. 为每个知识点补一行简短说明
5. 如果能确定最终会落到哪个 docs 主题文件，就填好 `领域`
6. 如果 `领域` 或 docs 分类目录不明确，先与用户确认

## 生命周期

```text
生成 todo.md -> 逐条完成 demo -> 对应行标记已完成
                                 ->
                        全部完成并经用户确认
                                 ->
                     按领域拆分迁移到 docs 分支主题文件
                                 ->
                             删除 todo.md
```

关键规则：

1. `todo.md` 在整轮学习过程中持续保留，不因单次 demo 完成而删除
2. 所有条目完成并且用户确认后，才迁移到 docs 分支
3. 即使 `todo.md` 使用多张表或多阶段结构，迁移时也不是原样复制整个文档，而是先将每行的 `领域` 映射到语义一致的主题文件，再分发到对应的 `xx.md`
4. 迁移到 docs 时，补上 `Commit` 和 `时间` 字段
5. 如果某个知识点之前已经存在于 docs 中，且这次写了新的 demo，则用最新 demo 覆盖该知识点的索引行
6. 这里的“新的 demo”仍然要拿新的 `dN` 编号，不能沿用旧知识点过去的编号
7. 如果一个 demo / MVP 形成了 `dN -> dN+...` 或 `mN -> mN+...` 的连续提交链，则迁移 docs 时要融合整条链判断最终索引
8. 默认使用 `dN` / `mN` 的 `学习主题` 和 `具体描述` 作为 docs 中的 `知识点` 与 `说明`
9. `Commit` 和 `时间` 默认指向这条连续提交链的最后一个补丁提交

## 示例

**示例 1：围绕同一个主题文件**

```markdown
# 学习计划 - Regex 基础必修

> 目标：系统掌握正则表达式常用核心能力，并形成可回溯 demo

| 领域 | 知识点 | 说明 | 时间 |
|------|--------|------|------|
| regex | 普通字符和元字符 | 基础字符匹配、特殊元字符含义 | - |
| regex | 量词 | 贪婪与非贪婪匹配 | - |
| regex | 分组 | 分组、捕获组、命名捕获、反向引用、非捕获组 | - |
```

**示例 2：围绕实战目标拆路线**

```markdown
# 学习计划 - 用 Rust 重构 bykcli Core

> 目标：完成 bykcli Core 的 Rust 重构所需基础能力储备

| 领域 | 知识点 | 说明 | 时间 |
|------|--------|------|------|
| rust | ownership and borrowing | ownership rules, borrowing, lifetimes | - |
| rust | error handling | Result, Option, custom errors | - |
| rust | trait and generics | abstraction, reusable interfaces | - |
```

**示例 3：按阶段拆成多张表，单表允许多个领域**

```markdown
# 学习计划 - 做一个 CLI + Web 的最小实验

## 学习目标

> 目标：完成一个带命令行入口和简单前端页面的最小可运行实验

## 第一阶段

| 领域 | 知识点 | 说明 | 时间 |
|------|--------|------|------|
| rust | command line parsing | clap basics, args, subcommands | - |
| js | event loop | macrotask, microtask, execution order | - |

## 第二阶段

| 领域 | 知识点 | 说明 | 时间 |
|------|--------|------|------|
| rust | error handling | Result, anyhow, custom messages | - |
| js | fetch api | request, response, error handling | - |
```
