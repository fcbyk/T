# Demo / MVP 代码规范

本规范用于 T 仓库中的 demo 与 MVP 代码产出。

目标不是追求“工业级大而全”，而是让代码满足以下要求：

1. 可以运行
2. 便于讲解
3. 能突出当前知识点
4. 便于在后续 commit 中继续演化

## 总原则

1. 优先最小可运行版本，不要预先设计过度抽象
2. 一个 demo 只突出一组核心知识点，避免主题漂移
3. 代码结构要让用户一眼看出入口、核心逻辑和实验点
4. 宁可少写辅助层，也不要让示例被样板代码淹没
5. 代码要服务于学习笔记本身，贴近代码的知识点优先写在注释里，README 更偏概述

## 形态选择

默认先判断本次 demo 属于单文件脚本还是标准项目结构。判断标准：主题本身是否依赖构建流程、脚手架或固定目录结构？否 → 单文件；是 → 项目结构。

> 详细形态特征与案例见 `references/directory-structure.md`。

## 领域映射

代码产出涉及的 `领域` 映射规则见 `references/docs-format.md` 的"领域映射规则"。

## 文件组织

1. 只创建本次 demo 真正需要的文件
2. 单文件脚本模式：**一个知识点 = 一个 app 文件**，按序号命名 `app1.rs`、`app2.rs`、`app3.rs` ...。每个文件覆盖一个独立知识点，保持轻量可读
3. 每个 app 文件自身即独立可运行的完整入口（含 `fn main()`）
4. 不要为了"像正式项目"而默认生成过多目录层级
5. 如果学习目标本身包含构建、脚手架或目录约定，应直接切换到对应项目结构
6. 如果是 MVP，目录可以比 demo 稍完整，但仍应保持克制
7. 目录生命周期相关规则遵循 `references/directory-structure.md`

## 代码风格

1. 变量、函数、模块命名应直观表达语义
2. 优先清晰的控制流，避免炫技写法
3. 函数职责尽量单一，便于单段讲解
4. 学习型 demo 默认应有足够注释，优先把笔记写在最贴近代码的位置
5. 注释不仅可以解释“为什么”，也可以直接承载知识点、概念说明、对比和生命周期分析
6. README 不适合承载的细碎学习笔记，优先写进注释

## Demo 特有要求

1. 代码中应保留清晰的实验入口，便于用户直接运行和修改
2. 如果有多组对比实验，命名应让差异一目了然
3. 如涉及边界情况，至少展示一个代表性例子
4. 如涉及错误处理，展示最能体现知识点的处理方式
5. 如果某段代码仅用于教学对比，应优先在注释中标明用途；必要时再在 README 里做总览说明

## MVP 特有要求

1. MVP 关注“能跑通一个完整最小场景”
2. 可以包含必要的基础结构，但不要引入当前阶段讲不清的复杂框架设计
3. 如果用户没有明确要求，不要默认加入鉴权、配置中心、复杂测试脚手架等扩展内容
4. 优先保证主路径可运行，再考虑增强项

## README 配合要求

README 不绑定代码文件，只做知识罗列。用表格展示知识点分布（`# / 知识点 / 说明`）。

> 详细规范见 `references/directory-structure.md`。

## 不推荐做法

以下写法应尽量避免：

1. 为了“看起来专业”而加入与主题无关的设计模式
2. 一次 demo 同时演示太多概念
3. 省略运行入口，导致用户拿到代码却不知道怎么验证
4. 本该贴近代码解释的知识点全堆到 README，导致读代码时信息断层
5. 生成大量样板文件，但核心知识点只有很少几行
