# 自主编码模式 README 规范

本规范用于 T 仓库中"用户自己写代码、AI 负责讲解与引导"的学习场景。

## 与 AI 写代码模式的区别

| 维度 | AI 写代码（能力二） | 用户写代码（能力六） |
|------|---------------------|----------------------|
| 知识笔记载体 | 源码注释 | README |
| README 角色 | 概述 + 运行说明 | 知识笔记 + 验证指引 |
| AI 产出的代码 | 完整可运行 | 说明性片段（对话中，不写入文件） |
| README 写入时机 | 与代码一起产出 | 用户说"更新 README"时触发 |

自主编码模式中，AI 的回答可以包含知识拆解表、概念讲解、验证清单和说明性代码片段；README 不需要照搬这些回答结构，只沉淀用户已经学习和验证过的知识点内容。

## README 结构

```markdown
# dN 知识点标题

> 一句话主题概述

## 知识点

### 知识点 A

（从对话中提炼出的知识点笔记：是什么、为什么重要、关键行为、常见误区或对比）

```rust
// 如果对话中的说明性代码片段对理解仍有帮助，可以保留精简片段
```

### 知识点 B

（继续记录另一个知识点）

## 自主验证清单

- [ ] 实现 X 并验证 Y
- [ ] 对比 A 和 B 的行为差异
- [ ] 尝试边界情况 Z

## 运行说明

（如果涉及特定运行方式或环境要求，写在这里；否则可省略）
```

## 各模块说明

### 首行标题

```markdown
# dN 知识点标题
```

格式与 AI 写代码模式一致，`dN` 编号与 commit message 中的编号对应。

### 知识点

README 的主体是知识点笔记。每个知识点使用独立小节，记录用户已经通过对话理解、并准备或已经通过代码验证的内容。

每个知识点小节建议包含：

1. **是什么** — 一两句话定义
2. **为什么重要** — 可选，但建议写上
3. **关键行为** — 用户编码时应观察到什么
4. **注意什么** — 常见坑点或与相似概念的对比
5. **代码片段** — 可选；只保留最能辅助理解的精简片段

不要在 README 中强制保留 AI 回答里的知识拆解表、完整讲解结构或所有验证建议。那些结构主要用于对话中的教学和引导。

代码片段示例格式：
````markdown
### 所有权移动

所有权移动表示值的所有者从一个变量转移到另一个变量。移动后，原变量不再可用。

```rust
let s1 = String::from("hello");
let s2 = s1;  // s1 的所有权移动到 s2
// println!("{}", s1);  // 编译错误：s1 已失效
```

常见的理解误区是把 move 当成浅拷贝...
````

关键规则：
- 代码片段是**讲解素材**，只展示与概念直接相关的部分，不追求可独立运行
- 片段注释应解释"为什么"，不只是"是什么"

### 自主验证清单

一份用户可以逐项勾选的动手练习列表。

规则：
- 每项对应一个具体的验证动作，而不是泛泛的"把概念 A 的代码跑一遍"
- 包含预期结果，让用户知道什么是"验证通过"
- 鼓励对比和边界测试

### 运行说明

如果用户代码需要特定的运行方式（比如需要编译参数、依赖安装、特定环境变量），写在这里。如果只需 `npm run appN` 或 `cargo run`，可省略。

## 增量追加规则

每次用户说"更新 README"时，按以下规则操作：

1. 回顾对话上下文，识别本次讨论的知识点
2. 检查 README 中是否已有这些知识点的章节
3. 已有 → 检查是否需要更新（用户可能有新理解）
4. 没有 → 在 README 的 `## 知识点` 下新增对应小节
5. 多个新知识点时，按讲解顺序逐一追加，每个知识点一个小节

追加时：
- 只追加本次新增或明显更新的知识点内容
- 如 README 有 `## 自主验证清单`，可按需追加与本次知识点直接相关的验证项
- 如没有验证清单，且本次不需要单独沉淀验证动作，可以不新增该模块

不要：
- 覆盖或删除之前的已沉淀内容
- 把多个无关知识点捏成一个标题
- 把 AI 回答中的知识拆解表、概念讲解结构、验证清单完整照搬进 README
- 在对话中讲完所有知识点后一次性追加一个巨大章节（应逐知识点追加）

## 对话到 README 的整理原则

从对话上下文提取内容写入 README 时：

1. **提炼而非复制** — 对话是口语化的，README 应精炼为书面化笔记
2. **按知识点沉淀** — 每个知识点对应一个小节，不强制保留 AI 回答的教学结构
3. **保留关键代码片段** — 对话中的示例代码如果仍有教学价值，精简后写入 README
4. **删掉冗余** — 对话中的追问、确认、感慨等自然语言交互不写入 README
5. **统一术语** — 同一个概念在整份 README 中使用一致的术语表达

## 生命周期

```
AI 讲解知识 → 用户编码验证 → 用户说"更新 README"
                                      ↓
                            AI 从对话提炼，增量追加到 README
                                      ↓
                            用户检查 README 内容
                                      ↓
                            用户提交（代码 + README 一起）
                                      ↓
                            README 随代码一起被下一个 commit 覆盖
                            （历史通过 git 回溯）
```

README 和代码始终在同一 commit 中，不分开提。这确保 checkout 到任意历史 commit 时，代码和对应的知识笔记始终同步。
