# docs 分支索引规范

本规范用于 T 仓库 `docs` 分支中的知识索引文件。

docs 分支的作用不是重复 README 正文，而是提供一份可回溯的知识目录：

1. 按主题文件汇总知识点
2. 通过 commit 链接跳回对应历史快照
3. 作为后续复习与回顾入口

## 文件结构

docs 分支按目录分类组织主题文件，例如：

```text
基础必修/regex.md
WEB前端/javascript.md
移动端开发/swift.md
```

其中：

1. 目录名表示上层分类，如 `基础必修`、`WEB前端`
2. 文件名 `xx.md` 表示一个主题文件
3. commit message 中的 `领域` 默认映射到这里的某个主题文件

### 领域映射规则（唯一定义源）

`领域`、docs 主题文件名、Markdown 标题可以不完全同名，但必须表达同一个主题，并保持映射稳定。

核心约束：
- `领域` 可以使用简称、别名或缩写（如 `js`、`ts`、`regex`）
- docs 文件名可以使用更完整的规范名称（如 `javascript.md`、`typescript.md`）
- 文件标题可以使用更自然的表达（如 `## JavaScript`）
- 只要三者语义一致，就视为同一个主题，应复用同一个主题文件
- 一旦某个主题已有固定 docs 文件名，后续沿用，不要因别名不同新建重复文件
- 若目标主题文件不存在，先确认是否只是已有主题文件的别名；只有确认不存在稳定映射时，才新建 `xx.md`

> 本节为"领域映射"规则的唯一定义源。其他 reference 文件遇到此规则时仅引用本节，不再重复表述。

## 单个主题文件格式

以 `regex.md` 为例：

```markdown
## Regular Expression

| 知识点 | 说明 | Commit | 时间 |
| ----- | ---- | ------ | ---- |
| 分组 | 分组、捕获组、命名捕获、反向引用、非捕获组 | [demo5] | 2026-05-19 |

[demo5]: https://github.com/fcbyk/T/tree/<commit-hash>
```

## 字段含义

| 字段 | 来源 | 说明 |
|------|------|------|
| `知识点` | commit 中的 `学习主题` | 对应一次 demo 的核心知识点 |
| `说明` | commit 中的 `具体描述` | 对知识点的展开目录或提纲 |
| `Commit` | commit hash 链接 | 指向当前索引所采用的最新提交快照 |
| `时间` | 最新提交时间 | 当前索引所指向那条最新提交的时间 |

补充说明：

1. `领域` 是主题标识，映射规则见上方"领域映射规则"
2. docs 表格中的 `知识点`、`说明` 默认使用中文表达
3. 如果某个概念很难用自然中文准确表达，或英文术语本身更稳定、更常用，可以直接保留英文

## 与 commit message 的对应关系

基准 commit:

```text
regex(d5): grouping

capturing groups, named groups, backreferences, non-capturing groups
```

如果后面紧接着还有连续补丁提交：

```text
regex(d5+): fix grouping example
regex(d5+): add named capture sample
```

对应 docs 行：

```markdown
| 分组 | 捕获组、命名捕获、反向引用、非捕获组 | [demo5] | 2026-05-21 |
```

## 编写要求

1. `知识点` 是索引标题，保持聚焦，不要写成句子
2. `说明` 是展开目录，简洁即可，不必写成长段说明
3. 更完整的原理、代码讲解和运行方式，写在对应 commit 快照中的 `README.md`
4. `Commit` 列统一使用引用式链接，如 `[demo5]`
5. commit message 即使使用英文，映射到 docs 时也应优先转成自然中文
6. 若直译后的中文不自然、不准确，或社区里更常用英文术语，可直接保留英文

### 字段选取规则

docs 索引行的四个字段选取，统一遵循下方"选字段步骤"，本节不再重复表述。

## 更新规则

1. 新增知识点时，在对应主题文件表格中追加一行
2. 若同一个知识点后来又产出新的 demo，则更新该知识点原有行，字段选取规则见下方"选字段步骤"
3. 若该 demo / MVP 后面还有连续的 `dN+` / `mN+`，同样遵循"选字段步骤"
4. 旧 demo 不在主索引表中重复保留，索引默认只维护该知识点的最新状态
5. 这里的"最新 demo"应使用新的 `dN` / `mN` 编号，因为它是一次新的 demo / MVP，不是旧提交的补丁
6. 领域映射规则见上方"领域映射规则"
7. 若 `领域` 能确定文件名但目录分类不明确，先与用户确认放到哪个目录
8. 不要把 README 正文整段复制到 docs 表格中
9. 更新 docs 前，先检查当前 `main` 是否已经提交干净；若未提交干净，先提醒用户处理
10. 不自动切换分支；默认先提示用户手动切换到 `docs` 分支，除非用户明确要求代为切换
11. 更新 docs 后，默认由用户自己检查内容并决定是否 git 提交

### 当用户指定更新某个 `xxx.md`

如果用户不是泛泛地说“更新 docs”，而是明确说要更新某个主题文件，例如 `javascript.md`，默认按以下流程处理：

1. 先从文件名提取主题标识，例如 `javascript.md` -> `javascript`
2. 先按这个主题标识查对应提交，也就是先查 `javascript(...)`
3. 如果按文件名查不到，再查该主题已知的稳定别名，例如 `javascript` 查不到再查 `js`
4. 如果别名有歧义，或者查出来的提交看起来不是同一个主题，先向用户确认要查哪个 `领域`
5. 不要因为用户点名了某个 `xxx.md`，就跳过别名映射判断；仍然要优先复用已存在的稳定映射

默认应从 `git log` 中整理出这些信息：

| 查询结果 | 用途 |
|---------|------|
| 提交信息 | 映射为 docs 表格中的 `知识点`、`说明` |
| 提交哈希值 | 生成 `Commit` 链接 |
| 提交时间 | 填写 docs 表格中的 `时间` |

其中“提交信息”通常指：

1. commit 标题中的 `学习主题`
2. commit 正文中的 `具体描述`

例如用户要求更新 `javascript.md` 时：

1. 默认先查 `javascript`
2. 如果没有结果，再查 `js`
3. 如果用户一开始就说明“这个文件对应提交里的 `js`”，则直接按 `js` 查

推荐查询思路示例：

```bash
# 先按文件名主题查
git log --grep='^javascript(' --oneline

# 如果没有结果，再按稳定别名查
git log --grep='^js(' --oneline
```

如果 AI 无法可靠判断该文件对应哪个提交领域，应先询问用户，而不是直接猜。

### 标准提示词模板

默认可直接使用下面这段提示：

```text
main 分支看起来已经提交干净了。请你先手动切换到 docs 分支；你切好之后我再继续帮你更新 docs 内容。更新完成后，你再自己检查并决定是否提交。
```

如果 `main` 还不干净，则先提示用户处理未提交改动，不继续 docs 流程。

## docs 索引如何从一串提交里选字段

当一个知识点对应的提交链形如：

```text
regex(d4): grouping
regex(d4+): fix example
regex(d4+): add named capture sample
```

默认按以下规则选字段：

| docs 字段 | 默认来源 | 说明 |
|----------|----------|------|
| `知识点` | `dN` / `mN` 的 `学习主题` | 基准提交定义主题 |
| `说明` | `dN` / `mN` 的 `具体描述` | 基准提交定义目录提纲 |
| `Commit` | 这条连续提交链最后一个提交 | 指向当前最新可回看的快照 |
| `时间` | 这条连续提交链最后一个提交时间 | 反映最近一次更新 |

### 选字段步骤

1. 先找到该知识点当前对应的基准提交，也就是 `dN` 或 `mN`
2. 判断后面是否紧接着存在连续的 `dN+` / `mN+`
3. 如果没有连续补丁提交：
   - `知识点` 取 `dN` / `mN` 的 `学习主题`
   - `说明` 取 `dN` / `mN` 的 `具体描述`
   - `Commit` 与 `时间` 指向 `dN` / `mN`
4. 如果有连续补丁提交：
   - `知识点` 仍默认取 `dN` / `mN` 的 `学习主题`
   - `说明` 仍默认取 `dN` / `mN` 的 `具体描述`
   - `Commit` 与 `时间` 指向最后一个 `dN+` / `mN+`
5. 如果后来不是补丁，而是重新写了一版新的 demo / MVP：
   - 切换到新的 `dN` / `mN` 作为基准提交
   - 旧链不再占主索引记录

### 例外情况

以下情况可以不完全照搬基准提交的 `说明`：

1. 基准提交的 `具体描述` 过长，不适合放到 docs 表格
2. 连续补丁提交让最终版本的重点发生了轻微偏移
3. 需要把多条零散的小修内容融合成更稳定的一行索引说明

但默认原则仍然是：

1. docs 的 `知识点` 与 `说明` 优先保持稳定
2. docs 不追着每个 `dN+` / `mN+` 的小改动频繁改标题
3. 详细变化看提交链与对应 `README.md`
